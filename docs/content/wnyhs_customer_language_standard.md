# WNYHS Customer Language Standard — REV01

## Purpose

This document defines the customer-facing language, page structure, CTA rules, and visual consistency standards for WNY Home Security pages.

Use this standard when updating:

* Homepage
* QR Landing
* Solutions pages
* Fit Check
* Estimate / callback flow
* Support
* Our Work / Example Projects
* Future customer-facing pages

The goal is consistency, clarity, trust, and conversion.

---

# Core Customer Message

WNY Home Security sells practical smart-property solutions, not generic hardware packages.

The customer should understand:

```text
What problem this solves
Why WNYHS is different
What happens next
How to contact us
```

# CTA Rules

Primary CTA:

```text
Request a Callback
```

Secondary CTA:

```text
Call/Text 716-201-0364
```

When both appear together, use:

```text
[ Request a Callback ]   OR   [ Call/Text 716-201-0364 ]
```

Rules:

* Do not use vague CTA labels.
* Do not use `Get Started`.
* Do not overload `Request a Free Estimate`.
* Use `On-site estimate` only when the customer is actually requesting a property visit.
* Fit Check is an optional helper, not a forced path.
* Avoid stacking multiple unrelated CTAs in the same panel.

# Page Structure Rules

Customer-facing pages should follow this pattern where possible:

```text
What this page is about
Why it matters
How WNYHS helps
What happens next
Trust / proof
Clear CTA
```

Panel rules:

* One heading per panel.
* No duplicate eyebrow + heading combinations like `SOLUTIONS` + `Solutions`.
* Strong panel separation.
* Consistent panel width.
* Consistent typography.
* Cards must have enough contrast to read as cards.
* No decorative filler symbols.
* No generic marketing slogans unless they say something specific.
* No orphan buttons.
* No back links unless the page truly needs one.

# Visual Consistency Rules

Use the current WNYHS homepage as the visual reference.

Apply:

* same panel width
* same heading hierarchy
* same card spacing
* same dark premium visual language
* same semantic tokens
* same button style
* same CTA row pattern

Do not:

* hardcode colors
* introduce a new design system
* make pages look like generic SaaS templates
* leave cards flat when they need contrast
* mix old and new naming systems

---

# QR Landing Rules

QR Landing should be simpler than the homepage.

Assume the visitor came from a physical sign or QR code.

The page should quickly confirm:

```text
You are in the right place.
We help with cameras, video doorbells, package theft, garage/property awareness, and smart home security.
You can request a callback or call/text now.
```

QR Landing should not:

* over-explain WNYHS
* show back links
* force Fit Check
* use generic package language
* bury the CTA
* introduce technical concepts

# Claims Guardrails

Do not claim:

```text
guaranteed prevention
guaranteed police response
guaranteed life safety
medical monitoring
never fails
always works
hack-proof
crime-proof
```

Preferred safe language:

```text
reduce risk
increase awareness
help you know
designed to keep working
built around your property
no required monthly contracts
```

---

# Final Rule

Every customer-facing page should pass this test:

```text
Would a normal homeowner understand what this means, why it matters, and what to do next?
```

If not, simplify the page before adding more content.
