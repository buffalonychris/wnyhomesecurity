## Current QR Routing Model

- Physical placards currently use one shared destination: `/qrlanding`.
- Runtime assumption for current campaign reporting: `/qrlanding` traffic is treated as placard-attributed unless evidence shows meaningful non-placard promotion.
- Route-level attribution is directional, not absolute identity-proof attribution.

## QRLanding Attribution Assumptions (Current)

1. `/qrlanding` is treated as the dedicated placard acquisition route.
2. Cloudflare request counts include non-human and non-page-view traffic.
3. Visitor-level directional inference is allowed only for campaign trend analysis.
4. Final performance decisions require multi-signal comparison (requests, visitors, form submissions, calls, booked quotes, closed jobs).

## Current Attribution Limitations

- No per-placard source parameter exists in the current shared QR destination.
- No runtime guarantee that all `/qrlanding` sessions came from physical scans (reloads, direct link shares, crawlers, and internal checks can appear).
- Cloudflare request telemetry includes assets/background traffic and should not be interpreted as unique humans.
- Early reported metrics are directional only and cannot independently prove conversion quality.

## Future Source-Parameter Evolution Model

Planned, documentation-only model for future bounded implementation:

- Introduce explicit source taxonomy (example: `src=placard`, `campaign=wny_local_qr`, optional `placement_id`).
- Keep `/qrlanding` as valid fallback route for legacy placards.
- Preserve backward compatibility for submissions without source parameters.
- Add governance rule: parameter introduction must not bypass `/api/lead-signal` intake boundary.

## requestId + Event Lifecycle Expectations

Expected event chain for QR funnel attribution:

1. `qrlanding_view` occurs when a visitor lands on `/qrlanding` (telemetry/event layer; current implementation status may vary).
2. `estimate_form_started` occurs when estimate capture interaction begins.
3. `estimate_form_submitted` occurs when submission is attempted to `/api/lead-signal`.
4. `/api/lead-signal` returns canonical server-generated `requestId` used for downstream correlation.

### requestId Persistence Expectations

- `requestId` authority remains server-side and is generated by `/api/lead-signal`.
- `requestId` should be retained in submission-level logs/records where available for traceability.
- If submission fails, surfaced `requestId` remains the operator/customer support correlation key.

### Timestamp Expectations

- Event records should capture UTC-capable timestamps.
- Ordering expectation: `qrlanding_view` <= `estimate_form_started` <= `estimate_form_submitted`.
- Submission timestamp and server-side `requestId` timestamp segment are expected to be directionally consistent.

### entryRoute Expectations

- `entryRoute` should preserve the originating route value (`/qrlanding`) when available.
- Downstream lead handling should treat `entryRoute=/qrlanding` as campaign-attribution context, not standalone proof of physical scan origin.

## Cloudflare Analytics Interpretation Guidance

### Requests vs Visitors

- **Requests**: all HTTP requests observed (HTML, assets, bot traffic, probes, reloads).
- **Visitors**: Cloudflare-estimated human/browser visitor metric with its own modeling/limits.
- Requests are expected to materially exceed visitors.

### Performance Telemetry vs Visitor Analytics

- Performance telemetry (request volume, status codes, cache behavior) is operational signal.
- Visitor analytics is directional campaign signal.
- Do not use performance telemetry counts as direct conversion denominators without normalization.

### Acceptable KPI Interpretation Rules

- Use 7-day and 28-day trend windows rather than single-day spikes.
- Treat `/qrlanding` request deltas as upper-bound attention signal.
- Tie campaign decisions to lead-stage outcomes, not request totals alone.

### Bot/Crawler Noise Considerations

- Assume bot/crawler/background noise exists in raw request metrics.
- Compare ratios over time (for example `estimate_form_submitted` per `/qrlanding` visitors) instead of raw request counts only.
- Use anomaly checks for sudden request surges without matching lead signals.

## Campaign KPI Model (Current Governance)

Primary KPI ladder:

1. Placard placements (inventory deployed).
2. `/qrlanding` traffic (directional top-of-funnel interest).
3. Estimate requests (lead submissions / `/api/lead-signal`).
4. Phone calls (inbound contact attempts).
5. Booked quotes (owner-confirmed scheduling outcomes).
6. Closed jobs (completed revenue outcomes).

Governance posture:

- Campaign health should be evaluated as conversion down the ladder, not any single metric.
- KPI ownership remains modular: runtime telemetry, lead intake, scheduling confirmation, and sales close-out are separate bounded systems.
