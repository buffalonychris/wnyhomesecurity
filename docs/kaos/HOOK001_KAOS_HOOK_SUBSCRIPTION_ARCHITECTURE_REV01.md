# HOOK001 KAOS Hook Subscription Architecture REV01

Status: Active KAOS foundation architecture
Task ID: HOOK001
Customer-facing: No
Implementation authority: No

## 1. Purpose

KAOS hooks are future governed subscribers to Codex lifecycle events and WNYHS workflow events.

This architecture defines how future hooks may observe events, evaluate completed work, produce candidate context, support KAOS intake, support RSI capture, and reduce manual administrative overhead. It does not implement hooks or automation.

Hooks are not events. Hooks are not authority. Hooks do not automatically implement decisions. Hooks must reduce manual reconciliation and operator overhead without creating side-work nightmares. Hook maturity is diagnostic, not a hard-stop gate, unless higher-authority governance, protected-system rules, active task scope, or operator decision requirements require stricter review.

## 2. Hook Definition

A KAOS hook is:

> A governed subscriber that may evaluate a Codex or WNYHS workflow event, produce structured context or candidate records, and route findings for operator review without independently creating authority or implementation work.

A hook can help identify what deserves review. It cannot decide that a candidate is approved, activate a task, merge a pull request, approve a deployment, or change protected systems.

## 3. Event vs Hook / Automation Distinction

| Concept | Meaning | Authority posture |
| --- | --- | --- |
| Event | Something happened. | Factual signal only. |
| Hook | Subscriber that evaluates the event. | Diagnostic or advisory unless bounded implementation later defines stricter handling. |
| Automation | Implemented execution of a hook or workflow response. | Requires separate bounded authorization. |
| Candidate Output | Non-authoritative intake generated by hook evaluation. | Must be reviewed before promotion. |
| Operator Review | Human approval, rejection, or routing decision. | Required when protected systems, authority, or scope decisions are implicated. |
| Repository Promotion | Bounded task or owner-document update. | Only path from candidate to durable authority. |

Events describe what happened. Hooks evaluate those facts. Automations execute approved responses. Candidate outputs remain non-authoritative until operator review and repository promotion occur through the normal task and owner-document process.

Important Codex hook notes for future implementation design:

- Multiple matching command hooks for the same event may run concurrently.
- Hooks may come from user, project, plugin, or managed config layers.
- Non-managed command hooks require review and trust before execution.
- Project-local hooks require a trusted project `.codex/` layer.
- Hook commands receive JSON on stdin.
- Hook output may provide context, block certain supported calls, or continue/stop certain flows depending on event.
- Hooks cannot be treated as complete enforcement boundaries.

## 6. Hook Priority / Review Levels

| Level | Meaning | Typical handling |
| --- | --- | --- |
| Informational | Helpful context or low-risk observation. | Record or inject only when useful. |
| Conditional | Review recommended when specific conditions are met. | Warn and route to owner doc or operator when relevant. |
| Required Review | Human/operator review is required before promotion or follow-up. | Stop follow-up until reviewed. |
| Protected-System Review | Protected systems, claims, secrets, runtime, deployment, or financial behavior may be affected. | Require owner docs, runtime contracts, and explicit authorization before action. |
| Blocking Guardrail | Clear unauthorized protected-system or destructive action is detected. | Block supported flow or require explicit operator authorization. |

Most hooks should be informational or conditional. Blocking hooks should be rare and limited to clear safety, protected-system, secret, destructive, or authority violations. Hooks should reduce human effort, not expand process overhead.

## 8. Hook Input / Output Rules

Allowed hook outputs:

- additional developer context
- warning message
- candidate KAOS intake
- candidate RSI record
- candidate decision record
- candidate relationship update
- candidate task recommendation
- validation warning
- protected-system warning
- operator decision request

Prohibited hook outputs:

- direct implementation authority
- automatic task activation
- automatic PR merge
- automatic deployment approval
- automatic owner-document rewrite
- automatic deletion or consolidation
- automatic protected-system modification

Unknown fields in hook-generated candidates must be marked `Unknown`, not inferred.

## 9. Trust and Safety Model

Future hook implementation must follow a trust model before any command hook is executed.

Rules:

- Hook trust review is required before command hooks run in normal WNYHS workflow.
- Project-local hooks require trusted project `.codex/` configuration before use.
- Managed hooks may be useful later, but they are not implemented by this task.
- Plugin-bundled hooks require provenance review and scope review before use.
- Hook source precedence must be documented before multiple user, project, plugin, or managed layers are combined.
- Hook hash changes require renewed review when trust depends on a reviewed command body or script body.
- Safe default behavior is to warn or decline rather than execute untrusted hook code.
- Hooks must not provide unsafe bypasses around repository governance, protected-system locks, or permission review.

`--dangerously-bypass-hook-trust` must not be part of normal WNYHS workflow.

## 10. Protected-System Hook Rules

Hooks touching or evaluating these areas require stricter rules:

- Stripe/payment/deposits/refunds
- scheduling
- HubSpot/CRM lifecycle
- customer-facing claims
- warranty/support commitments
- installed asset lifecycle
- dashboard/customer validation
- procurement/inventory financial behavior
- runtime/API behavior
- legal/compliance-sensitive artifacts
- Cloudflare deployment configuration
- secrets/environment variables
- dependencies/package-lock
- destructive git operations

Stricter rules mean the hook needs a defined owner document, input/output contract, trust posture, validation expectations, and explicit operator review path before implementation.

Protected-system hooks should prefer warnings and operator decision requests over automated action. Blocking behavior is appropriate only when a clear unauthorized protected-system, secret, destructive, or dependency/package-lock risk is detected and the future hook implementation supports a safe block.

## 11. Hook Maturity Levels

| Level | Name | Meaning |
| --- | --- | --- |
| H-L0 | Concept only | Hook idea exists without durable owner record. |
| H-L1 | Candidate documented | Hook is named with candidate purpose and status. |
| H-L2 | Owner document defined | Owner document and authority boundary are identified. |
| H-L3 | Input/output contract defined | Inputs, outputs, prohibited outputs, trust, and failure handling are specified. |
| H-L4 | Script design approved | Implementation design is approved through a bounded task. |
| H-L5 | Implemented through bounded task | Hook code/config exists only after explicit implementation authorization. |
| H-L6 | Trusted and validated | Hook has trust review, validation evidence, and safe failure behavior. |
| H-L7 | Automation-ready / self-governing | Hook can support governed automation under mature owner docs and review rules. |

Hook maturity is diagnostic, not blocking. It helps identify readiness and risk without turning every candidate into process overhead.

## 12. Example Hook Flows

### Example A - Docs-only task closeout

```text
Stop event
-> KAOS Intake Hook
-> RSI Hook
-> Context Efficiency Hook
-> candidate outputs only
-> operator review
-> future task if approved
```

### Example B - Source/runtime task

```text
PreToolUse / PermissionRequest
-> Protected System Guardrail Hook
-> warn/block only if unauthorized protected files or commands are involved
-> PostToolUse review
-> Stop review
```

### Example C - Deployment review

```text
PR_MERGED / DEPLOYMENT_COMPLETED equivalent
-> Deployment QA Hook candidate
-> Sites/Playwright future routing
-> QA evidence
-> RSI or task candidate if failure observed
```

## 13. Prohibited Behavior

This architecture must not:

- implement hooks
- create hooks.json
- create config.toml hooks
- write hook scripts
- install Playwright
- configure Sites
- change source/runtime files
- change site behavior
- automatically create tasks
- automatically promote knowledge to authority
- automatically merge PRs
- automatically approve deployments
- treat every hook finding as a blocker
- create excessive hook-admin overhead
- mark future backlog tasks DONE
- change HubSpot, Stripe/payment, scheduling, quote, catalog, dashboard, SEO, runtime, route, CSS, dependency, package-lock, environment, secret, public-copy, or Cloudflare behavior
