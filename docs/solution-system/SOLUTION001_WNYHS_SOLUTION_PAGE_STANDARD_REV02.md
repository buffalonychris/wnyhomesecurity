# SOLUTION001_WNYHS_SOLUTION_PAGE_STANDARD_REV02

## Purpose

This document establishes the formal WNY Home Security Solution Page Standard.

The purpose of this standard is to eliminate page drift as the WNY Home Security solution catalog expands.

This standard governs:

* Solution page structure
* Visual hierarchy
* Typography
* Card layout
* CTA standards
* Copy standards
* Image standards
* Internal linking standards

REV02 adds the approved solution image system for `solution-hero-image` and `solution-sample-image`.

This standard governs:

* /solutions/senior-safety
* /solutions/water-protection
* /solutions/family-awareness
* /solutions/vacation-homes

Following visual approval, this standard may be promoted to a locked governance document.

---

# Core Principle

Every solution page must answer:

"Can WNY Home Security help solve this homeowner problem in a practical, local, and low-pressure way?"

Solution pages are not:

* Hardware catalogs
* Generic service pages
* Company overview pages
* Trust-heavy marketing pages

Solution pages are:

* Problem-focused
* Outcome-oriented
* Educational
* Conversion-supportive

# Required Solution Page Structure

All future solution pages must follow this structure.

## Section 1 — Hero

Required:

* Eyebrow
* Problem-focused H1
* Supporting paragraph
* Trust pill row
* Primary CTA
* Secondary CTA
* Optional hero image

Trust pills:

* Local Smart Property Solutions
* No Required Monthly Fees
* Customer-Owned Equipment
* Professionally Installed & Locally Supported

---

## Section 2 — Who This Helps

Purpose:

Allow the visitor to self-identify.

Examples:

* Adult children supporting aging parents
* Homeowners worried about frozen pipes
* Families managing arrivals and packages
* Seasonal property owners

Focus on people and situations.

Not hardware.

---

## Section 3 — Problem / Solution / Outcome

Required:

Three-column layout.

### Problem

Describe concern.

### Solution

Describe planning approach.

### Outcome

Describe awareness, confidence, and practical control.

Avoid guarantees.

Use:

* Helps you know sooner
* Helps reduce uncertainty
* Supports better decisions

---

## Section 4 — Western New York Context

Purpose:

Demonstrate local relevance.

Include:

* WNY winters
* Spring thaw
* Older homes
* Basements
* Detached garages
* Seasonal properties
* Family routines

Avoid fabricated local stories.

---

## Section 5 — Possible Pieces Of The Plan

Purpose:

Illustrate possible solution components.

Format:

* Pills
* Tiles
* Short labels

Examples:

Water Protection:

* Water Sensor Placement
* Temperature Awareness
* Sump Awareness
* Mechanical Room Planning
* Alert Routing

Family Awareness:

* Door Awareness
* Garage Awareness
* Package Visibility
* Driveway Visibility
* Smart Lock Planning

Do not imply fixed scope.

Do not imply pricing.

Required.

Allowed labels:

* Common Situation
* Typical Situation
* How This Might Look

Do not:

* Use customer names
* Invent towns
* Invent outcomes

## Section 7 — Next Step

Purpose:

Convert without pressure.

Preferred CTA hierarchy:

Keep CTA language consistent.

# Solution Image System Standard

Every solution page should use two distinct image types:

1. `solution-hero-image`
2. `solution-sample-image`

They must perform different jobs.

Core distinction:

* Hero image = emotional/lifestyle outcome.
* Sample image = situation + relevant hardware + focused WNYHS-style awareness panel.

Do not let both images become dashboards.

Do not let both images become lifestyle photos.

---

## Solution Hero Image Standard

Purpose:

The `solution-hero-image` communicates the homeowner outcome.

It should answer:

* Is this my situation?
* Can I picture this helping me?
* Does this feel calm, local, practical, and trustworthy?

Required qualities:

* Human-centered scene
* Realistic homeowner/property context
* Positive outcome
* Professional residential photography style
* Calm, premium, local, practical tone
* Relevant solution context

May include subtle relevant hardware naturally present in the scene, such as:

* Pendant
* Watch wearable
* Camera
* Video doorbell
* Smart lock
* Leak sensor
* Temperature sensor

Hardware must be supporting context only.

Forbidden in hero images:

* Dashboard UI
* Full control panels
* Text overlays
* Headlines
* Marketing copy
* CTA buttons
* Trust pills
* Pricing
* Feature lists
* Fake alert overlays
* Fake testimonial language
* Before/after claims
* Medical emergency scenes
* Panic/fear imagery
* Sci-fi interfaces

Hero image formula:

Situation + Outcome + Property Context

Hero images should not explain the system.

Hero images should make the visitor feel the benefit.

---

## Solution Sample Image Standard

Purpose:

The `solution-sample-image` appears in or near the Example Scenario / How This Might Look section.

It should answer:

* What would WNYHS actually put together?
* What relevant devices might be involved?
* What would the customer see or know?
* How would the solution reduce uncertainty?

Required composition:

### Zone A: Situation

Approximate visual weight: 60-70%.

Show the real homeowner scenario.

Examples:

* Senior and loved one in a home setting
* Family arrivals / package / driveway activity
* Utility room / basement water concern
* Seasonal or second property context

### Zone B: Relevant Hardware

Approximate visual weight: 10-15%.

Show only a small number of representative devices.

This is not a product catalog.

Examples:

Aging In Place:

* Pendant
* Activity watch
* Discreet motion or entry sensor if appropriate

Water Protection:

* Leak sensor
* Temperature sensor
* Water shutoff valve or sump-related device if appropriate

Family Awareness:

* Video doorbell
* Smart lock
* Door/garage sensor

Vacation Home:

* Camera
* Leak sensor
* Temperature sensor
* Smart access device

### Zone C: Awareness Panel

Approximate visual weight: 20-25%.

Show one focused WNYHS-style dashboard card or panel.

The panel should communicate:

* Status
* Activity
* Awareness
* Simple notifications
* What the homeowner knows sooner

It should not be a full Home Assistant dashboard screenshot unless explicitly approved.

It should look practical and achievable, not sci-fi.

Sample image formula:

Situation + Relevant Hardware + Awareness Panel + Outcome

---

## Image Relationship Rule

The two images must not duplicate each other.

Hero Image:

* Emotional
* Lifestyle
* Outcome
* No dashboard

Sample Image:

* Educational
* Practical
* Hardware + dashboard panel
* Scenario support

Hero image explains why the visitor wants the solution.

Sample image explains how the solution might help.

---

## Image Claims Guardrails

Do not allow image standards that imply:

* Guaranteed protection
* Guaranteed prevention
* Guaranteed detection
* Medical monitoring
* Emergency response
* Fall detection unless explicitly scoped and supportable
* Insurance savings
* Real customer installation proof
* Verified testimonial proof
* Fabricated local stories
* Fabricated customer outcomes

Images may be illustrative only.

Alt text and documentation language must avoid implying images are real customer installations unless separately verified.

# Approval Workflow

1. Add this document to repo.
2. Execute SOLUTION001-A.
3. Visually review all four pages.
4. Approve structure and styling.
5. Create SOLUTION001-B.
6. Add the approved solution image system.
7. Visually review any future generated or replacement production images against REV02 before wiring them into the site.

Future solution pages must follow the current approved revision of this standard.
