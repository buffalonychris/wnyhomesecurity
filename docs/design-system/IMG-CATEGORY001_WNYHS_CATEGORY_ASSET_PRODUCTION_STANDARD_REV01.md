## 1. Purpose

IMG-CATEGORY001 defines the official production, approval, organization, and usage standards for category-level visual assets used throughout WNY Home Security.

This standard governs:

- category landing page heroes
- lower-right branded hardware clusters in hero images
- dashboard images
- mobile dashboard images
- WNYHS Core panel images
- reveal/explainer images
- solution thumbnails
- routine thumbnails
- future AI-assisted image generation workflows

This document is governance only. It does not authorize creating, replacing, wiring, publishing, or generating image assets.

## 3. Core Visual Principles

Category assets must be:

- realistic and residential
- bright enough to understand quickly
- category-specific
- calm, premium, and local in tone
- useful at the size where they appear
- compatible with the WNYHS dark/gold visual system
- claim-safe and free of panic-based imagery

Assets must not:

- pretend to be real customer installations unless separately verified
- use marketing slogans inside images
- depend on tiny text for meaning
- use fictional sci-fi dashboards
- use unrelated generic tech art
- use dark unreadable scenes
- introduce a separate visual system from the approved Home Automation reference page

## 4. Required Asset Classes

Each future category asset package should include, when scoped by a separate production task:

- hero image
- lower-right category hardware cluster for the hero image
- dashboard image
- mobile dashboard image
- WNYHS Core category image
- reveal/explainer image or image pair
- solution thumbnails
- routine thumbnails
- image manifest

Not every asset class must be used on every page, but every produced asset must be assigned to one governed class before approval.

## 5. Hero Image Standard

Hero images must contain a primary lifestyle scene that establishes the category in a real home context.

Required:

- primary residential lifestyle scene
- clear category connection
- lower-right area reserved for category-relevant hardware
- no dependency on text overlays
- realistic lighting and perspective
- safe crop behavior across desktop, tablet, and mobile

The lower-right hardware area must feel integrated into the image. It may show a small cluster of relevant devices, but it must not turn the hero into an ad graphic.

Hero images must not include:

- marketing slogans
- floating labels
- oversized ad graphics
- fake alert banners
- pricing
- CTA buttons
- dense dashboard UI
- unrelated hardware

## 6. Category Hardware Identity System

Each category has a hardware identity system. Hardware shown in category hero clusters, Core images, reveal images, or thumbnails must be associated with the category.

Home Security examples:

- camera
- video doorbell
- smart lock
- door/window sensor

Home Automation examples:

- smart switch
- presence sensor
- motion sensor
- smart plug

Aging in Place examples:

- presence sensor
- activity sensor
- safety device
- caregiver-oriented awareness device

Home Safety examples:

- leak sensor
- freeze sensor
- environmental sensor
- utility awareness device

Home Lighting examples:

- smart dimmer
- motion sensor
- lighting controller
- pathway lighting device

These examples are not product approvals. Final hardware selection, exact product depiction, and part-level fit require separate bounded catalog or asset approval.

## 7. Hardware Branding Standard

Hardware may include subtle WNYHS branding when it supports category identity.

Allowed:

- small WNYHS mark
- subtle device badge
- restrained brand color accent
- clean device placement in the lower-right hero area

Not allowed:

- large logo stickers
- slogans on devices
- billboard-style overlays
- fake certification marks
- branding that overwhelms the lifestyle scene

Branding must support recognition without implying an official hardware manufacturer relationship unless that relationship is separately authorized.

## 8. Dashboard Authenticity Standard

Dashboard images must look realistic and usable.

Required:

- category-relevant controls
- status cards
- awareness panels
- plain homeowner labels
- believable spacing and hierarchy
- readable primary content at intended display size

Dashboard images must not look like:

- fictional sci-fi UI
- cybersecurity screens
- random charts
- generic tech art
- dense operator consoles
- unreadable developer panels
- unrelated analytics screens

Use outcome-oriented labels such as status, activity, awareness, check-in, condition status, scene, routine, zone, device, and schedule.

## 10. Aging in Place Dashboard Accessibility Standard

Aging in Place dashboard images have a stricter accessibility and tone requirement.

Required:

- white or light theme
- clear and concise layout
- large readable controls
- calm spacing
- approachable labels
- minimal technical detail

Avoid:

- dark mode
- dense dashboards
- tiny controls
- complex graphs
- technical layouts
- alarming colors or language

The image should feel calm, reassuring, and easy to use for families and caregivers.

## 12. Thumbnail Standards

Thumbnail images must be:

- bright
- instantly recognizable
- grounded in a real home context
- built around one clear focal subject
- readable at 150-220px width
- visually consistent across the set

Thumbnail images must not be:

- UI screenshots
- infographics
- dark unreadable scenes
- dense hardware catalogs
- text-heavy images
- unrelated lifestyle filler

Solution thumbnails should show the outcome or situation. Routine thumbnails should show the routine context. Neither should rely on labels inside the image.

## 13. Visual Parity Standard

All category assets must maintain parity with the approved Home Automation category page reference implementation.

Parity includes:

- section image rhythm
- image hierarchy
- dashboard/mobile/routine relationship
- thumbnail brightness and recognizability
- WNYHS Core category relevance
- dark/gold site compatibility
- calm residential image tone
- no separate category-specific visual language

Future category assets may be distinct by subject matter, but they must feel like they belong to the same WNYHS system.

## 14. File Naming Standard

Asset filenames must be stable, descriptive, lowercase where practical, and specific to category and purpose.

Recommended pattern:

```text
<category-slug>-<asset-role>.<ext>
<category-slug>-<solution-slug>-thumb.<ext>
<category-slug>-<routine-slug>-thumb.<ext>
```

Examples:

```text
home-safety-hero.jpg
home-safety-dashboard.jpg
home-safety-mobile-dashboard.jpg
home-safety-core.jpg
home-safety-leak-awareness-thumb.jpg
home-lighting-pathway-routine-thumb.jpg
```

Avoid:

- vague filenames
- temporary export names
- spaces
- version clutter in production filenames
- names that describe forbidden or unsupported claims

## 15. Folder Structure

Expected category asset folder:

```text
public/images/category-pages/<category-slug>/
```

Expected subfolders may be used only when a production task authorizes larger packages:

```text
public/images/category-pages/<category-slug>/source/
public/images/category-pages/<category-slug>/approved/
public/images/category-pages/<category-slug>/rejected/
```

Production tasks should keep source, approved, and rejected files organized so application code only references approved production assets.

## 16. Export Standards

Recommended export standards:

- photographic hero images: JPG or PNG, sRGB
- thumbnails: JPG, sRGB, quality 85-92
- dashboard images: PNG or high-quality JPG where UI legibility matters
- mobile dashboard images: PNG or high-quality JPG
- source files: retain separately when production task scope allows
- transparency: only when the image role requires it

Every export must be checked at the rendered size. Dashboard images must remain readable. Thumbnails must remain recognizable at 150-220px width.

## 17. Image Manifest Requirement

Every future category image production task must include an image manifest unless the bounded task explicitly narrows scope.

The manifest should record:

- asset filename
- asset class
- category
- intended page or section
- source or generation method
- approval status
- reviewer or approval note
- rejected/replaced status when relevant
- accessibility notes
- claim-safety notes

The manifest is evidence for approval. It is not an implementation instruction by itself.

## 18. Approval Checklist

Before a category asset is approved:

- [ ] Asset class is identified.
- [ ] Category relevance is clear.
- [ ] Hardware is associated with the category.
- [ ] Hero hardware, if present, stays in the lower-right area.
- [ ] Dashboard visuals show realistic controls, status cards, or awareness panels.
- [ ] Aging in Place dashboards use a light, accessible theme.
- [ ] WNYHS Core images are category-specific.
- [ ] Thumbnails remain recognizable at 150-220px width.
- [ ] Visual parity with Home Automation is preserved.
- [ ] File naming follows the standard.
- [ ] Folder placement follows the standard.
- [ ] Export format is appropriate.
- [ ] Manifest entry exists.
- [ ] No image asset implies unsupported outcomes.
- [ ] No protected system or runtime change is bundled with image approval.

## 19. Rejected Asset Workflow

Rejected assets must not be wired into application code.

Rejected assets should be:

- moved to a rejected folder when such a folder is in scope
- recorded in the manifest
- labeled with the rejection reason
- replaced only through a bounded production task

Common rejection reasons:

- wrong category hardware
- unreadable dashboard
- dark or unclear thumbnail
- generic tech imagery
- unsupported claim implication
- visual mismatch with the Home Automation reference page
- hardware branding too prominent

## 20. Future Catalog Integration

Future catalog integration may connect approved category assets to canonical category, solution, routine, and hardware records.

Future integration must preserve:

- source-of-truth catalog boundaries
- product approval gates
- hardware status review
- claim-safe image language
- separation between documentation approval and runtime implementation

IMG-CATEGORY001 does not authorize catalog schema changes, runtime catalog changes, product approval, or image wiring.
